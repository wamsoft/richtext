# RichText ライブラリ設計資料

## 1. 概要

吉里吉里用のリッチテキストレンダリングライブラリ。minikin によるテキストレイアウト、FreeType によるグリフパス化、thorvg によるベクター描画を組み合わせて、高品質なテキスト描画を実現する。

### 1.1 主要機能

- **1行テキストレンダリング**: シンプルな1行テキストの高速描画
- **パラグラフレンダリング**: 矩形領域内での複数行テキスト描画（左揃え・中央・右揃え対応）
- **多言語対応**: 複数言語の混在、RTL（右から左）テキスト対応
- **フォントフォールバック**: 複数フォントを束ねて文字カバレッジを補完
- **絵文字対応**: カラー絵文字を含むテキスト描画
- **スタイル指定**: フォント、サイズ、色、スペーシングなどのスタイル制御
- **装飾描画**: ストローク（縁取り）と塗り（Fill）を複数回重ねる装飾処理
- **インラインタグ**: HTMLライクなタグでテキスト途中のスタイル変更
- **CJK ロケール対応**: 同一コードポイントでもロケール別に異なる字形を描画

## 2. 使用ライブラリと役割

| ライブラリ | 役割 | 管理方法 |
|-----------|------|---------|
| **minikin** | テキストレイアウトエンジン（行分割、グリフ配置座標計算） | git submodule (`ext/minikin`) |
| **thorvg** | ベクターグラフィックス描画（SW エンジン） | git submodule (`ext/thorvg`) |
| **FreeType** | フォント読み込み、グリフのアウトラインパス取得 | vcpkg |
| **harfbuzz** | 複雑なテキスト整形（minikin 内部で使用） | minikin に内包 |
| **ICU** | Unicode 処理、行分割ルール（minikin 内部で使用） | minikin に内包 |
| **zlib / libpng** | PNG 出力（サンプル用） | vcpkg |

### 2.1 データフロー

```
テキスト入力 (UTF-16)
        ↓
[TagParser] タグ解析
  - HTMLライクタグをパース
  - StyleRun 配列 + Appearance 配列に変換
        ↓
[minikin] テキストレイアウト
  - フォント選択（フォールバック処理）
  - MeasuredText で文字幅計算
  - breakIntoLines で行分割
  - グリフID・配置座標の算出
        ↓
[FreeType] グリフパス化
  - グリフIDからアウトラインパス取得
  - カラー絵文字はビットマップ取得
        ↓
[thorvg] ベクター描画
  - Appearance に従い Stroke → Fill の順で描画
  - ビットマップへのレンダリング
        ↓
出力ビットマップ (ARGB)
```

## 4. 主要クラス

### 4.1 FontManager

フォントの登録・管理を行うシングルトンクラス。

- `registerFont(path, name)` — フォントファイルを名前付きで登録
- `createCollection(names)` — フォント名リスト（優先度順）からフォールバックコレクション作成
- `registerLocale(locale)` — ロケール文字列を登録しIDを返す（例: `"ja_JP"`, `"zh_Hant"`）

### 4.2 FontFace

`minikin::MinikinFont` を継承。FreeType フォントフェイスをラップ。

- `getGlyphPath()` — グリフのアウトラインパスを thorvg 形式で取得
- `getGlyphBitmap()` — カラー絵文字用ビットマップ取得
- `isColorGlyph()` — カラー絵文字かどうか判定

### 4.3 TextStyle / DrawStyle / Appearance

**TextStyle**: フォント、サイズ、ウェイト、イタリック、字間、ロケールなどの論理スタイル。`applyTo()` で `minikin::MinikinPaint` に変換。

**DrawStyle**: 個別の描画操作（Fill または Stroke）。色、オフセット、ストローク幅など。

**Appearance**: 複数の DrawStyle を組み合わせた描画外観。描画順序は配列順で、**Stroke を先（下）、Fill を後（上）** に配置することで縁取り効果を実現。

### 4.4 TextLayout

1行テキストのレイアウト。`minikin::Layout` を使用。

### 4.5 ParagraphLayout

複数行テキストのレイアウト。`minikin::MeasuredText` + `breakIntoLines` を使用。

- `getRunWidth(start, end)` — MeasuredText の文字幅を合算。TextRenderer でのセグメント配置に使用し、行分割とレンダリングの幅を一致させる。

### 4.6 GlyphRenderer

グリフ単位の描画。thorvg キャンバスにパスまたはビットマップを描画。

- フェイクイタリック: `FontFakery::isFakeItalic()` が true の場合、`renderPath` でパスポイントにシアー変換（`skewX = -0.25`）を適用。パスキャッシュはシアー前の形状を保持し描画時に変換する。

### 4.7 TextRenderer

描画の統合インタフェース。

- `drawText()` — 1行テキスト描画
- `drawParagraph()` — パラグラフ描画
- `drawStyledText()` — TagParser 結果を受けてスタイル付きテキスト描画

### 4.8 TagParser

HTMLライクなインラインタグを解析し、`StyleRun` + `Appearance` の配列に変換。

対応タグ:
- `<font size=... weight=... spacing=...>` フォント変更
- `<color value=0xAARRGGBB>` / `<color r=... g=... b=... a=...>` 色変更
- `<b>` / `<strong>` 太字
- `<i>` / `<em>` 斜体
- `<u>` 下線
- `<s>` / `<strike>` / `<del>` 取り消し線
- `<ruby text='...'>` ルビ
- `<sup>` 上付き / `<sub>` 下付き
- `<outline color=... width=...>` 縁取り
- `<shadow color=... x=... y=...>` 影
- `<style name='...'>` 名前付きスタイル参照
- `<br>` 改行 / `<sp width=...>` スペース

エスケープ: `&lt;`, `&gt;`, `&amp;`, `&quot;`, `&apos;`, `&nbsp;`, `&#nnnn;`, `&#xhhhh;`

## 6. 描画順序の設計

Appearance 内の DrawStyle は配列順に描画される。正規化された描画順序:

1. オフセット付き Fill（影）を最初に描画（最背面）
2. Stroke（縁取り）を次に描画
3. メインの Fill（本体色）を最後に描画（最前面）

この順序により、影が最も奥に、縁取りがその上に、塗りが最前面に表示される。`TagParser` の `applyColorTag` / `applyOutlineTag` / `applyShadowTag` は `rebuildDrawOrder()` で自動的にこの順序を保証する。

## 7. ルビ（振り仮名）の設計

### 7.1 概要

ルビはベーステキストの上部に小さな文字で読み仮名等を表示する機能。minikin にはルビのネイティブサポートがないため、描画層（`TextRenderer`）で独自に実装する。

### 7.2 タグ書式

```
<ruby text='ふりがな'>漢字</ruby>
```

- `text` 属性: ルビとして表示する文字列（UTF-8/UTF-16）
- タグ内のテキスト: ルビが付くベーステキスト

### 7.3 パース

`TagParser` が `<ruby>` タグを解析し、`TextSpan` に以下を設定:

| フィールド | 型 | 内容 |
|-----------|-----|------|
| `hasRuby` | `bool` | `true` |
| `rubyText` | `std::u16string` | ルビ文字列 |

### 7.4 レイアウト・描画

`TextRenderer::drawStyledText` 内で、ルビ付きセグメントの描画後にルビを追加描画する。

**ルビレイアウト処理:**

1. ルビ用 `TextStyle` を生成（ベーススタイルの `fontSize × rubyScale`）
2. ルビ文字列を `TextLayout` で独立にレイアウト
3. ルビを水平方向にベーステキスト上で中央揃え配置
4. ルビを垂直方向にベーステキストの ascent 直上に配置

**配置計算:**

```
rubyX = segmentX + (segmentWidth - rubyWidth) / 2
rubyY = baseY + baseAscent + rubyDescent
```

- `baseAscent` は負値（上方向）なので、ベースラインから上に移動
- `rubyDescent` は正値なので、ルビの下端がベーステキストの上端に接する

### 7.5 パラメータ

`TagParser::ParseOptions` で調整可能:

| パラメータ | デフォルト | 説明 |
|-----------|-----------|------|
| `rubyScale` | `0.5` | ルビのフォントサイズ倍率（ベースサイズに対する比率） |

### 7.6 行間との関係

ルビは行の ascent・descent や行間計算に影響を与えない。ルビが正しく表示されるためには、呼び出し側で十分な行間を確保する必要がある。

**1行目の上方はみ出し:** パラグラフ領域の1行目ではルビがベーステキストの ascent より上に描画されるため、領域の上端からはみ出す。はみ出しを避けるには、描画領域の y 座標をルビ分（`fontSize × rubyScale` 程度）下げて調整する。

### 7.7 ルビの重なり回避

隣接する文字に別々の `<ruby>` タグを付けると、ルビテキストがベーステキストより長い場合にルビ同士が重なる。

```
<!-- NG: ルビが重なる可能性あり -->
<ruby text='とうきょう'>東京</ruby><ruby text='タワー'>塔</ruby>

<!-- OK: ベーステキストをまとめて1つの ruby タグで囲む -->
<ruby text='とうきょうタワー'>東京塔</ruby>
```

ルビ幅がベーステキスト幅を超える場合のレイアウト調整（字間拡張や圧縮）は行わないため、ルビが長くなる場合はベーステキストを広くとるか、1つの `<ruby>` タグにまとめて対処する。

### 7.8 制限事項

- **行またぎ非対応**: ルビ付きテキストが行分割で複数行にまたがった場合、ルビは描画されない（セグメントがスパン全体をカバーする場合のみ描画）
- **ルビ幅の行分割への影響なし**: ルビがベーステキストより長い場合でも、行分割計算にルビ幅は考慮されない（はみ出す可能性がある）
- **グループルビのみ**: 1文字ずつルビを対応させるモノルビには非対応（グループ全体に対して一括配置）

## 8. 双方向テキスト（Bidi）の設計

### 8.1 概要

Unicode Bidirectional Algorithm（UBA）に基づき、LTR（左から右）と RTL（右から左）のテキストを混在して描画する。minikin 内部の ICU `ubidi` を使用してテキストのラン分割と視覚順序の決定を行う。

### 8.2 Bidi モード

`TextStyle::bidi` で段落方向を指定する。minikin の `Bidi` 列挙値に対応:

| 値 | ICU レベル | 動作 |
|---|---|---|
| `Bidi::LTR` | 0 | 段落方向を LTR に強制 |
| `Bidi::RTL` | 1 | 段落方向を RTL に強制 |
| `Bidi::DEFAULT_LTR` | `UBIDI_DEFAULT_LTR` | 最初の強い文字で段落方向を決定。強い文字がなければ LTR |
| `Bidi::DEFAULT_RTL` | `UBIDI_DEFAULT_RTL` | 最初の強い文字で段落方向を決定。強い文字がなければ RTL |

### 8.3 DEFAULT_LTR / DEFAULT_RTL の注意

`DEFAULT_LTR` と `DEFAULT_RTL` はいずれも「最初の強い方向性文字」で段落方向を決定する。違いは強い文字が**存在しない**場合のフォールバックのみ。

例: テキスト `"Hello مرحبا World"` の場合、最初の強い文字は `H`（LTR）であるため、`DEFAULT_LTR` でも `DEFAULT_RTL` でも段落方向は LTR になる。明示的に RTL 段落にするには `Bidi::RTL` を使用する。

### 8.4 CJK 文字の Bidi クラス

CJK 文字（漢字・ひらがな・カタカナ）の Unicode bidi クラスは **"L"（Left-to-Right）** である。これは Latin 文字と同じ扱いとなる。

RTL 段落内での動作:
- **ラン（ブロック）の配置順** は段落方向に従って入れ替わる（例: `[日本語][العربية][日本語]` → RTL では `[日本語][العربية][日本語]` の並び順が逆転）
- **CJK ラン内部の文字順** は常に LTR のまま維持される（「日本語」が「語本日」にはならない）

これは Unicode 標準に基づく正しい動作である。日本語の右から左表記は伝統的な縦書きや看板等に限られ、Unicode の水平テキスト処理では LTR として扱われる。

### 8.5 制限事項

- **`FORCE_LTR` / `FORCE_RTL` は使用不可**: minikin 内部でアサーション違反となるため、`Bidi::LTR` / `Bidi::RTL` を使用すること
- **段落分割との関係**: `\n` で分割された各サブ段落は独立した bidi コンテキストを持つ
