# RichText ライブラリ設計資料

## 1. 概要

多言語・装飾対応のリッチテキストレンダリングライブラリ。minikin によるテキストレイアウト、FreeType によるグリフパス化、thorvg によるベクター描画を組み合わせて、高品質なテキスト描画を実現する。

### 1.1 主要機能

- **1行テキストレンダリング**: シンプルな1行テキストの高速描画
- **パラグラフレンダリング**: 矩形領域内での複数行テキスト描画（左揃え・中央・右揃え対応）
- **多言語対応**: 複数言語の混在、RTL（右から左）テキスト対応
- **フォントフォールバック**: 複数フォントを束ねて文字カバレッジを補完
- **絵文字対応**: カラー絵文字を含むテキスト描画
- **スタイル指定**: フォント、サイズ、色、スペーシングなどのスタイル制御
- **装飾描画**: ストローク（縁取り）と塗り（Fill）を複数回重ねる装飾処理
- **インラインタグ**: HTMLライクなタグでテキスト途中のスタイル変更
- **CJK ロケール対応**: 同一コードポイントでもロケール別に異なる字形を描画

## 2. 使用ライブラリと役割

| ライブラリ | 役割 | 管理方法 |
|-----------|------|---------|
| **minikin** | テキストレイアウトエンジン（行分割、グリフ配置座標計算） | git submodule (`ext/minikin`) |
| **thorvg** | ベクターグラフィックス描画（SW エンジン） | git submodule (`ext/thorvg`) |
| **FreeType** | フォント読み込み、グリフのアウトラインパス取得 | vcpkg |
| **harfbuzz** | 複雑なテキスト整形（minikin 内部で使用） | minikin に内包 |
| **ICU** | Unicode 処理、行分割ルール（minikin 内部で使用） | minikin に内包 |
| **zlib / libpng** | PNG 出力（サンプル用） | vcpkg |

### 2.1 データフロー

```
テキスト入力 (UTF-16)
        ↓
[TagParser] タグ解析
  - HTMLライクタグをパース
  - StyleRun 配列 + Appearance 配列に変換
        ↓
[minikin] テキストレイアウト
  - フォント選択（フォールバック処理）
  - MeasuredText で文字幅計算
  - breakIntoLines で行分割
  - グリフID・配置座標の算出
        ↓
[FreeType] グリフパス化
  - グリフIDからアウトラインパス取得
  - カラー絵文字はビットマップ取得
        ↓
[thorvg] ベクター描画
  - Appearance に従い Stroke → Fill の順で描画
  - ビットマップへのレンダリング
        ↓
出力ビットマップ (ARGB)
```

## 4. 主要クラス

### 4.1 FontManager

フォントの登録・管理を行うシングルトンクラス。

#### フォントローダー

ライブラリ自体はファイルシステムへの直接アクセスを行わない。利用側がフォントデータの読み込み関数を登録する。

- `setFontDataLoader(loader)` — バッファ方式ローダーを登録。`FontDataBuffer`（`shared_ptr<vector<uint8_t>>`）を返す関数を設定する。**優先的に使用される。**
- `setFontStreamLoader(loader)` — FreeType ストリーム方式ローダーを登録。`FT_Stream` を返す関数を設定する。バッファローダーが未登録または読み込みに失敗した場合のフォールバックとして使用される。

フォント登録時（`registerFont` / `registerVariableFont`）に、登録されたローダーが呼び出されてフォントデータを取得する。

```cpp
// 使用例: ファイルシステムから読み込むバッファローダー
fm.setFontDataLoader([](const std::string& name) -> richtext::FontDataBuffer {
    std::ifstream file("./data/" + name, std::ios::binary | std::ios::ate);
    if (!file) return nullptr;
    auto size = file.tellg();
    file.seekg(0, std::ios::beg);
    auto buffer = std::make_shared<std::vector<uint8_t>>(size);
    if (!file.read(reinterpret_cast<char*>(buffer->data()), size))
        return nullptr;
    return buffer;
});

// フォント登録（ローダーにはファイル名が渡される）
fm.registerFont("NotoSans-Regular.ttf", "sans");
```

#### ローダー方式の選択指針

| 方式 | 用途 | 特徴 |
|------|------|------|
| **バッファ方式** | 通常利用、アーカイブ内フォント | フォントデータ全体をメモリに保持。FreeType は `FT_OPEN_MEMORY` でバッファを直接参照（内部コピーなし）。minikin / harfbuzz もバッファに直接アクセス可能 |
| **ストリーム方式** | 巨大フォントの遅延読み込み | FreeType の `FT_OPEN_STREAM` を使用。必要なテーブルのみを読み込むため初期メモリ消費が少ない。ただし `GetFontData()` が nullptr を返すため、minikin / harfbuzz のテーブル直接参照は FreeType 経由になる |

#### フォント管理 API

- `registerFont(fileName, name, index)` — フォント登録。`fileName` はローダーに渡される識別子、`name` は `createCollection` 等で使用する登録名
- `registerVariableFont(fileName, name, weight, italic, index)` — バリアブルフォントをウェイト・スタイル指定で登録
- `createCollection(names)` — フォント名リスト（優先度順）からフォールバックコレクション作成
- `registerLocale(locale)` — ロケール文字列を登録しIDを返す（例: `"ja_JP"`, `"zh_Hant"`）

### 4.2 FontFace

`minikin::MinikinFont` を継承。FreeType フォントフェイスをラップ。

2種類のコンストラクタを持ち、FontManager のローダーから呼び出される:

- `FontFace(name, shared_ptr<vector<uint8_t>>, index)` — バッファ方式。データの寿命は `shared_ptr` で管理
- `FontFace(name, FT_Stream, index)` — ストリーム方式。FontFace がストリームの所有権を取得

主要メソッド:

- `getGlyphPath()` — グリフのアウトラインパスを thorvg 形式で取得
- `getGlyphBitmap()` — カラー絵文字用ビットマップ取得
- `isColorGlyph()` — カラー絵文字かどうか判定
- `isVariableFont()` — バリアブルフォントかどうか判定
- `setVariations()` — バリエーション軸（wght, wdth 等）の値を設定

### 4.3 TextStyle / DrawStyle / Appearance

**TextStyle**: フォント、サイズ、ウェイト、イタリック、字間、ロケールなどの論理スタイル。`applyTo()` で `minikin::MinikinPaint` に変換。

**DrawStyle**: 個別の描画操作（Fill または Stroke）。色、オフセット、ストローク幅など。

**Appearance**: 複数の DrawStyle を組み合わせた描画外観。描画順序は配列順で、**影 Fill → Stroke → 通常 Fill** の順に配置する。`set*` / `add*` メソッドが自動的にレイヤー位置を管理する。

### 4.4 TextLayout

1行テキストのレイアウト。`minikin::Layout` を使用。

### 4.5 ParagraphLayout

複数行テキストのレイアウト。`minikin::MeasuredText` + `breakIntoLines` を使用。

- `getRunWidth(start, end)` — MeasuredText の文字幅を合算。TextRenderer でのセグメント配置に使用し、行分割とレンダリングの幅を一致させる。

### 4.6 GlyphRenderer

グリフ単位の描画。thorvg キャンバスにパスまたはビットマップを描画。

- フェイクイタリック: `FontFakery::isFakeItalic()` が true の場合、`renderPath` でパスポイントにシアー変換（`skewX = -0.25`）を適用。パスキャッシュはシアー前の形状を保持し描画時に変換する。
- フェイクボールド: `FontFakery::isFakeBold()` が true の場合、`renderPath` でグリフパスに同色ストローク（フォントサイズの 1/24 幅）を重ねて太字をシミュレートする。

### 4.7 StyledLayout

タグ付きテキストのレイアウト処理を分離したクラス。`TextRenderer::drawStyledText()` が内部で行っていたタグ解析・行分割・セグメント構築を事前計算し、結果を保持する。

**主要メソッド:**

- `layout(text, maxWidth, maxHeight, hAlign, vAlign, styles, appearances, lineSpacing)` — レイアウト実行

**取得可能な情報:**

- `getParsed()` — `TagParser::ParseResult`（スパン情報を含む）
- `getParagraphLayout()` — `ParagraphLayout`（行情報）
- `getLineLayouts()` — 行ごとのセグメントレイアウト配列（`SegmentLayout` に TextLayout・measuredWidth・spanIdx 等を格納）
- `getTotalGlyphCount()` — 総グリフ数
- `getTotalCharCount()` — 総文字数（ユニーク charIndex 数）
- `getHAlign()` / `getVAlign()` — アライン設定
- `getMaxWidth()` / `getMaxHeight()` — レイアウト領域サイズ

**用途:**

1. レイアウトを事前計算し、`TextRenderer::drawStyledLayout()` で描画（maxGlyphs による逐次表示対応）
2. `TextureAtlas::addStyledLayout()` でグリフをアトラスに登録し、`getCopyRects()` でコピー矩形を生成
3. `drawStyledText()` は内部で `StyledLayout` を生成して `drawStyledLayout()` を呼び出す

### 4.8 TextRenderer

描画の統合インタフェース。

- `drawText()` — 1行テキスト描画
- `drawParagraph()` — パラグラフ描画
- `drawStyledText()` — TagParser 結果を受けてスタイル付きテキスト描画（内部で StyledLayout を使用）
- `drawStyledLayout()` — 事前計算済み StyledLayout の描画（maxGlyphs による逐次表示対応）

### 4.9 TextureAtlas

グリフを事前レンダリングしてテクスチャに格納し、コピー矩形配列で表示順に転送する。

- `addLayout()` — TextLayout のグリフをアトラスに追加
- `addParagraphLayout()` — ParagraphLayout のグリフをアトラスに追加
- `addStyledLayout()` — StyledLayout のグリフをアトラスに追加（各セグメントのスタイル・外観を自動参照）
- `commit()` — アトラスをテクスチャに書き込み
- `getCopyRects()` — TextLayout / ParagraphLayout / StyledLayout から表示用コピー矩形配列を生成（maxGlyphs 対応）

### 4.10 TagParser

HTMLライクなインラインタグを解析し、`StyleRun` + `Appearance` の配列に変換。

対応タグ:
- `<font size=... weight=... spacing=... width=... face=...>` フォント変更（`face` で名前付きコレクションを参照可）
- `<color value=0xAARRGGBB [add]>` / `<color r=... g=... b=... a=... [add]>` 色変更
- `<b>` / `<strong>` 太字
- `<i>` / `<em>` 斜体
- `<u>` 下線
- `<s>` / `<strike>` / `<del>` 取り消し線
- `<ruby text='...'>` ルビ
- `<sup>` 上付き / `<sub>` 下付き
- `<outline color=... width=... [add]>` 縁取り
- `<shadow color=... x=... y=... [add]>` 影
- `<style name='...'>` 名前付きスタイル参照
- `<br>` 改行 / `<sp width=...>` スペース

`add` 属性（値不要のフラグ）を指定すると、既存のレイヤーを置換せず追加する（多段縁取り等）。詳細は `タグ仕様.md` を参照。

エスケープ: `&lt;`, `&gt;`, `&amp;`, `&quot;`, `&apos;`, `&nbsp;`, `&#nnnn;`, `&#xhhhh;`

## 6. 描画順序の設計

Appearance 内の DrawStyle は配列順に描画される。`set*` / `add*` メソッドが管理する正規化された描画順序:

1. オフセット付き Fill（影）を最初に描画（最背面）
2. Stroke（縁取り）を次に描画
3. メインの Fill（本体色）を最後に描画（最前面）

この順序により、影が最も奥に、縁取りがその上に、塗りが最前面に表示される。

`add` 属性付きタグでは既存レイヤーを保持したまま追加することで、多段縁取りや多段影を実現できる。

## 7. ルビ（振り仮名）の設計

### 7.1 概要

ルビはベーステキストの上部に小さな文字で読み仮名等を表示する機能。minikin にはルビのネイティブサポートがないため、描画層（`TextRenderer`）で独自に実装する。

### 7.2 タグ書式

```
<ruby text='ふりがな'>漢字</ruby>
```

- `text` 属性: ルビとして表示する文字列（UTF-8/UTF-16）
- タグ内のテキスト: ルビが付くベーステキスト

### 7.3 パース

`TagParser` が `<ruby>` タグを解析し、`TextSpan` に以下を設定:

| フィールド | 型 | 内容 |
|-----------|-----|------|
| `hasRuby` | `bool` | `true` |
| `rubyText` | `std::u16string` | ルビ文字列 |

### 7.4 レイアウト・描画

`TextRenderer::drawStyledText` 内で、ルビ付きセグメントの描画後にルビを追加描画する。

**ルビレイアウト処理:**

1. ルビ用 `TextStyle` を生成（ベーススタイルの `fontSize × rubyScale`）
2. ルビ文字列を `TextLayout` で独立にレイアウト
3. ルビを水平方向にベーステキスト上で中央揃え配置
4. ルビを垂直方向にベーステキストの ascent 直上に配置

**配置計算:**

```
rubyX = segmentX + (segmentWidth - rubyWidth) / 2
rubyY = baseY + baseAscent + rubyDescent
```

- `baseAscent` は負値（上方向）なので、ベースラインから上に移動
- `rubyDescent` は正値なので、ルビの下端がベーステキストの上端に接する

### 7.5 パラメータ

`TagParser::ParseOptions` で調整可能:

| パラメータ | デフォルト | 説明 |
|-----------|-----------|------|
| `rubyScale` | `0.5` | ルビのフォントサイズ倍率（ベースサイズに対する比率） |

### 7.6 行間との関係

ルビは行の ascent・descent や行間計算に影響を与えない。ルビが正しく表示されるためには、呼び出し側で十分な行間を確保する必要がある。

**1行目の上方はみ出し:** パラグラフ領域の1行目ではルビがベーステキストの ascent より上に描画されるため、領域の上端からはみ出す。はみ出しを避けるには、描画領域の y 座標をルビ分（`fontSize × rubyScale` 程度）下げて調整する。

### 7.7 ルビの重なり回避

隣接する文字に別々の `<ruby>` タグを付けると、ルビテキストがベーステキストより長い場合にルビ同士が重なる。

```
<!-- NG: ルビが重なる可能性あり -->
<ruby text='とうきょう'>東京</ruby><ruby text='タワー'>塔</ruby>

<!-- OK: ベーステキストをまとめて1つの ruby タグで囲む -->
<ruby text='とうきょうタワー'>東京塔</ruby>
```

ルビ幅がベーステキスト幅を超える場合のレイアウト調整（字間拡張や圧縮）は行わないため、ルビが長くなる場合はベーステキストを広くとるか、1つの `<ruby>` タグにまとめて対処する。

### 7.8 制限事項

- **行またぎ非対応**: ルビ付きテキストが行分割で複数行にまたがった場合、ルビは描画されない（セグメントがスパン全体をカバーする場合のみ描画）
- **ルビ幅の行分割への影響なし**: ルビがベーステキストより長い場合でも、行分割計算にルビ幅は考慮されない（はみ出す可能性がある）
- **グループルビのみ**: 1文字ずつルビを対応させるモノルビには非対応（グループ全体に対して一括配置）

## 8. 双方向テキスト（Bidi）の設計

### 8.1 概要

Unicode Bidirectional Algorithm（UBA）に基づき、LTR（左から右）と RTL（右から左）のテキストを混在して描画する。minikin 内部の ICU `ubidi` を使用してテキストのラン分割と視覚順序の決定を行う。

### 8.2 Bidi モード

`TextStyle::bidi` で段落方向を指定する。minikin の `Bidi` 列挙値に対応:

| 値 | ICU レベル | 動作 |
|---|---|---|
| `Bidi::LTR` | 0 | 段落方向を LTR に強制 |
| `Bidi::RTL` | 1 | 段落方向を RTL に強制 |
| `Bidi::DEFAULT_LTR` | `UBIDI_DEFAULT_LTR` | 最初の強い文字で段落方向を決定。強い文字がなければ LTR |
| `Bidi::DEFAULT_RTL` | `UBIDI_DEFAULT_RTL` | 最初の強い文字で段落方向を決定。強い文字がなければ RTL |

### 8.3 DEFAULT_LTR / DEFAULT_RTL の注意

`DEFAULT_LTR` と `DEFAULT_RTL` はいずれも「最初の強い方向性文字」で段落方向を決定する。違いは強い文字が**存在しない**場合のフォールバックのみ。

例: テキスト `"Hello مرحبا World"` の場合、最初の強い文字は `H`（LTR）であるため、`DEFAULT_LTR` でも `DEFAULT_RTL` でも段落方向は LTR になる。明示的に RTL 段落にするには `Bidi::RTL` を使用する。

### 8.4 CJK 文字の Bidi クラス

CJK 文字（漢字・ひらがな・カタカナ）の Unicode bidi クラスは **"L"（Left-to-Right）** である。これは Latin 文字と同じ扱いとなる。

RTL 段落内での動作:
- **ラン（ブロック）の配置順** は段落方向に従って入れ替わる（例: `[日本語][العربية][日本語]` → RTL では `[日本語][العربية][日本語]` の並び順が逆転）
- **CJK ラン内部の文字順** は常に LTR のまま維持される（「日本語」が「語本日」にはならない）

これは Unicode 標準に基づく正しい動作である。日本語の右から左表記は伝統的な縦書きや看板等に限られ、Unicode の水平テキスト処理では LTR として扱われる。

### 8.5 制限事項

- **`FORCE_LTR` / `FORCE_RTL` は使用不可**: minikin 内部でアサーション違反となるため、`Bidi::LTR` / `Bidi::RTL` を使用すること
- **段落分割との関係**: `\n` で分割された各サブ段落は独立した bidi コンテキストを持つ

## 9. 絵文字フォントの設計

### 9.1 概要

絵文字フォントには大きく分けて3種類の形式がある。本ライブラリは CBDT と COLRv1 のカラー絵文字、およびモノクロアウトライン絵文字に対応する。

| 形式 | 例 | 特徴 | 対応方法 |
|------|-----|------|---------|
| **CBDT/CBLC** | NotoColorEmoji.ttf | 固定サイズビットマップ（PNG）を内包 | FreeType の `FT_LOAD_COLOR` で取得 |
| **COLRv1** | Noto-COLRv1.ttf | ベクターベースのペイントグラフ（DAG） | 自前レンダリング（FreeType API + thorvg） |
| **モノクロ** | NotoEmoji-Regular.ttf | 通常のアウトラインフォント | 通常のグリフパス描画 |

### 9.2 カラー絵文字の判定

`FontFace::isColorGlyph()` で判定する。FreeType の `FT_Load_Glyph` に `FT_LOAD_COLOR` フラグを渡し、結果のビットマップが `FT_PIXEL_MODE_BGRA` であるか、または `FT_Get_Color_Glyph_Paint` が成功する（COLRv1 テーブルを持つ）場合にカラーグリフとみなす。

### 9.3 CBDT ビットマップ絵文字

NotoColorEmoji.ttf は 128×128 ピクセルの固定サイズ PNG ビットマップを内包する。

**取得手順:**

1. `FT_Select_Size(face, 0)` で固定サイズを選択
2. `FT_Load_Glyph` に `FT_LOAD_COLOR` を指定（`FT_LOAD_RENDER` は不要）
3. `FT_Bitmap` から BGRA データを RGBA に変換

**スケーリング:**

ビットマップは固定サイズ（`available_sizes[0].height` = 128px）で取得される。要求フォントサイズに合わせるスケール係数を `GlyphBitmap::strikeHeight` に格納し、描画時に `fontSize / strikeHeight` でスケーリングする。

**ベースライン位置:**

FreeType の `bitmap_left`（bearingX）と `bitmap_top`（bearingY）をスケール係数で乗算して描画位置を決定する。

### 9.4 COLRv1 ベクター絵文字

Noto-COLRv1.ttf は COLRv1 テーブルにベクターベースのペイントグラフを格納する。FreeType 2.13+ はペイントグラフの走査 API を提供するが、自動レンダリングモジュールは含まない。そのため `FontFace::renderCOLRv1Glyph()` で自前レンダリングを行う。

**レンダリング手順:**

1. `FT_Set_Pixel_Sizes` でピクセルサイズを設定
2. `FT_Get_Color_Glyph_Paint` でルートペイントを取得（`FT_COLOR_INCLUDE_ROOT_TRANSFORM` を指定し、デザイン単位→ピクセル変換行列を含める）
3. `FT_Get_Color_Glyph_ClipBox` でバウンディングボックスを取得し、一時描画バッファを確保
4. 一時 `tvg::SwCanvas` を作成し、ペイントグラフを再帰走査
5. 各 `PaintGlyph` ノードで `FT_Load_Glyph(FT_LOAD_NO_SCALE)` によりデザイン単位のアウトラインを取得
6. アウトラインを thorvg パスに変換し、ルートトランスフォーム（親行列）で座標変換
7. 子ペイント（Solid / LinearGradient / RadialGradient）に応じて塗り情報を設定
8. レンダリング後、ARGB バッファを RGBA に変換して `GlyphBitmap` に格納

**座標系の整合:**

- グリフアウトライン: `FT_LOAD_NO_SCALE` でデザイン単位（フォント単位）のまま取得
- ルートトランスフォーム: `FT_COLOR_INCLUDE_ROOT_TRANSFORM` に含まれる行列がデザイン単位→ピクセル単位への変換を担当
- 内部トランスフォーム（Translate / Scale / Rotate / Skew）: デザイン単位空間で適用され、親行列と合成

**対応ペイントタイプ:**

| ペイントタイプ | 説明 |
|--------------|------|
| `COLR_LAYERS` | 複数レイヤーの SRC_OVER 合成 |
| `GLYPH` | グリフアウトラインを形状として使用 |
| `SOLID` | 単色塗り（CPAL パレット参照） |
| `LINEAR_GRADIENT` | 線形グラデーション |
| `RADIAL_GRADIENT` | 放射グラデーション |
| `COLR_GLYPH` | 別のカラーグリフを参照 |
| `TRANSFORM` | アフィン変換 |
| `TRANSLATE` | 平行移動 |
| `SCALE` | 中心点指定のスケーリング |
| `ROTATE` | 中心点指定の回転 |
| `SKEW` | シアー変換 |
| `COMPOSITE` | 合成（backdrop → source 順で描画） |

### 9.5 フォント登録の推奨構成

```cpp
// カラー絵文字（CBDT ビットマップ）
fontManager.registerFont("NotoColorEmoji.ttf", "emoji");

// カラー絵文字（COLRv1 ベクター）
fontManager.registerFont("Noto-COLRv1.ttf", "emoji-colr");

// モノクロ絵文字
fontManager.registerFont("NotoEmoji-Regular.ttf", "emoji-mono");
```

フォントコレクションでは絵文字フォントをフォールバック末尾に配置する:

```cpp
auto collection = fontManager.createCollection({"sans", "ja", "emoji"});
```

### 9.6 制限事項

- **COLRv1 の COMPOSITE**: 現在は backdrop → source を単純に重ね描きしており、Porter-Duff 合成モードには未対応
- **COLRv1 のグラデーション座標**: グラデーション座標はデザイン単位のまま使用しているため、一部の複雑なグラデーション絵文字で描画が不正確になる場合がある
- **CBDT のサイズ**: 固定サイズ（128px）からのスケーリングのため、大サイズでの表示時にジャギーが発生する
- **COLRv0**: COLRv0（単純レイヤー方式）には非対応。COLRv0 のみのフォントではフォールバックされない

## 10. バリアブルフォントの設計

### 10.1 概要

バリアブルフォント（OpenType Variable Font）は1つのフォントファイルに複数の軸（weight, width, italic 等）を持ち、連続的なスタイル変化が可能なフォント形式。本ライブラリは以下の3軸に対応する。

| 軸タグ | 名称 | TextStyle フィールド | 範囲（一般的） | 備考 |
|--------|------|---------------------|----------------|------|
| `wght` | Weight | `fontWeight` | 100–900 | 太さ |
| `ital` | Italic | `italic` | 0 or 1 | イタリック |
| `wdth` | Width | `fontWidth` | 50–200 (%) | 字幅 |

各軸について、バリアブルフォントの軸が利用可能な場合はそれを使い、利用不可能な場合はフェイク処理（ストローク太字・シアー変換・水平スケール）で代替する。

### 10.2 アーキテクチャ

**フォント登録:**

1. `FontFace` コンストラクタで `FT_Get_MM_Var` によりバリアブルフォントの軸情報を取得し、`axes_` に格納。`wdth` 軸がある場合は `hasWdthAxis_`, `wdthMin_`, `wdthMax_`, `wdthDefault_` も保存
2. `FontFace::setVariations()` で `FT_Set_Var_Design_Coordinates` を呼び出し、FreeType のフォントフェイスに軸値を設定
3. `FontManager::registerVariableFont()` で同じフォントファイルから異なるウェイト・スタイルの FontFace を複数作成。`wght` 軸と `ital` 軸を設定
4. `FontManager::createCollection()` が同じ名前の全 FontEntry を1つの `minikin::FontFamily` にまとめ、`Font::Builder::setWeight().setSlant()` でウェイトとスラントを明示指定

**フォント選択（minikin 自動）:**

5. minikin が `TextStyle::fontWeight` と `TextStyle::italic` に基づいて最適な FontFace を自動選択
6. 要求に合致する FontFace がない場合、minikin が `FontFakery` フラグ（`isFakeBold()`, `isFakeItalic()`）を設定

**使用例:**

```cpp
// ウェイト別・スタイル別に登録（Upright 6ウェイト + Italic 6ウェイト）
for (uint16_t w : {100, 300, 400, 500, 700, 900}) {
    fm.registerVariableFont("NotoSans-VF.ttf", "sans-vf", w);           // Upright
    fm.registerVariableFont("NotoSans-Italic-VF.ttf", "sans-vf", w, true); // Italic
}
auto col = fm.createCollection({"sans-vf", "emoji"});

style.fontWeight = 700;        // → wght=700 の FontFace が選択される
style.italic = true;           // → ital=1 の FontFace が選択される
style.fontWidth = 75.0f;       // → wdth 軸 75% で描画（condensed）
```

### 10.3 wght 軸（ウェイト）とフェイクボールド

**バリアブルフォント wght 軸:**

`registerVariableFont()` でウェイトごとに FontFace を作成し、minikin のフォント選択に任せる。minikin は `fontWeight` に最も近いウェイトの FontFace を選択する。

**フェイクボールド（フォールバック）:**

バリアブルフォントや Bold ウェイトのフォントが登録されていない場合、minikin が `FontFakery::isFakeBold()` フラグを自動設定する（要求ウェイト ≥ 600 かつ実際のウェイトとの差 ≥ 200）。

`GlyphRenderer` はこのフラグを検出すると、グリフパスに同色ストローク（フォントサイズの 1/24 幅）を重ねて太字をシミュレートする。

```
品質の優先度（高→低）:
1. バリアブルフォント wght 軸 — フォント設計者が意図した太さ
2. ウェイト別フォントファイル — Bold 用に設計された別ファイル
3. フェイクボールド — ストローク太らせ（セリフの潰れや字間変化あり）
```

### 10.4 ital 軸（イタリック）とフェイクイタリック

**バリアブルフォント ital 軸:**

`registerVariableFont()` の `italic=true` 引数で、Italic 用 FontFace（`ital` 軸=1、`FontStyle::Slant::ITALIC`）を登録する。minikin が `TextStyle::italic=true` のとき自動的に Italic FontFace を選択し、実際のイタリック字形が使用される。

イタリック軸を持つ別ファイル（例: `NotoSans-Italic[wdth,wght].ttf`）を同じ名前で登録することで、Upright と Italic の両方のフォントが1つの FontFamily にまとまる。

**フェイクイタリック（フォールバック）:**

Italic FontFace が登録されていない場合、minikin が `FontFakery::isFakeItalic()` フラグを設定する。

`GlyphRenderer` はこのフラグを検出すると、グリフパスに X 方向のシアー変換（`skewX = -0.25`、約14度の右傾斜）を適用してイタリックをシミュレートする。

```
品質の優先度（高→低）:
1. バリアブルフォント ital 軸 — フォント設計者が意図したイタリック字形
2. イタリック専用フォントファイル — 別ファイルで提供
3. フェイクイタリック — シアー変換（字形は直立のまま傾けるだけ）
```

> **注意**: CJK フォント（NotoSansJP 等）は一般にイタリック軸を持たないため、`italic=true` を指定するとフェイクイタリックが適用される。

### 10.5 wdth 軸（字幅）とフェイク幅

**TextStyle::fontWidth:**

`TextStyle::fontWidth` でフォント幅をパーセント指定する（100.0 = 通常幅）。CSS の `font-stretch` に相当する。

| 値 | CSS 相当 |
|----|----------|
| 50 | ultra-condensed |
| 75 | condensed |
| 100 | normal |
| 125 | expanded |
| 150 | extra-expanded |

**描画時の処理（GlyphRenderer）:**

`fontWidth` が 100 以外の場合、以下のロジックで wdth 軸とフェイクスケールを組み合わせる:

```
requestedWidth = style.fontWidth

if (フォントに wdth 軸あり、範囲 [min, max]):
    clampedWidth = clamp(requestedWidth, min, max)
    font->applyWidth(clampedWidth)           // wdth 軸を設定
    fakeScaleX = requestedWidth / clampedWidth  // 軸で足りない分
else:
    fakeScaleX = requestedWidth / 100.0      // 全てフェイク
```

例: NotoSans VF（wdth 軸 62.5–100）の場合:

| fontWidth | wdth 軸値 | fakeScaleX | 説明 |
|-----------|-----------|------------|------|
| 40% | 62.5 | 0.64 | 軸限界 + フェイク縮小 |
| 80% | 80 | 1.0 | 軸のみで対応 |
| 100% | 100 | 1.0 | 通常（変更なし） |
| 150% | 100 | 1.5 | 軸限界 + フェイク拡大 |

**レイアウト時の処理（MinikinPaint + GetHorizontalAdvance）:**

`TextStyle::applyTo` で `paint.fontWidth` を設定する。`fontWidth` は minikin の `MinikinPaint` に追加したフィールドで、`FontFace::GetHorizontalAdvance` 内で参照される。

`GetHorizontalAdvance` は fontWidth に基づき wdth 軸を設定してからグリフをロードし、軸で対応できない残りの fake 分を advance に乗算して返す。これにより行分割（MeasuredText）とグリフ配置（Layout）の両方で fontWidth が正確に反映される。

**キャッシュ:**

minikin の `LayoutCacheKey` と `GlyphRenderer` のグリフキャッシュキーの両方に `fontWidth` を含めることで、異なる幅のレイアウト・グリフが正しくキャッシュされる。

```
品質の優先度（高→低）:
1. バリアブルフォント wdth 軸 — フォント設計者が意図した字幅変化
2. フェイク幅（水平スケール） — 字形を均一に伸縮（ストロークの太さも変化する）
```

### 10.6 FontFace の wdth 軸 API

```cpp
// wdth 軸の有無と範囲を取得
bool getWidthAxisRange(float& minWidth, float& maxWidth) const;

// wdth 軸の値を設定（他の軸は維持）
void applyWidth(float width) const;
```

`applyWidth()` は `const` メソッドだが、内部で FT_Face の可変状態を変更する。シングルスレッド前提の設計。
